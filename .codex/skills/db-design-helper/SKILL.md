# DB Design Helper

## Overview

この Skill は、`flowlet` のテーブル論理設計を整理するときに使う。
PostgreSQL 固有最適化よりも、`Java` のエンティティへ起こしやすい設計を優先する。

## Workflow

1. 入力を確認する
- 要件、ユースケース、既存ドメイン、既存テーブル案を確認する。
- 入力が曖昧な場合は、いきなり DDL へ進まず、まずテーブル責務の候補を整理する。

2. テーブル責務を切る
- エンティティ化を前提に、1 テーブル 1 責務を意識する。
- 多対多や履歴系は、中間テーブルや独立テーブルを優先する。

3. 論理設計を作る
- テーブルごとに、論理名、物理名、主キー、カラム、型、NULL 可否、制約を整理する。
- テーブルコメントとカラムコメントを日本語で付ける。

4. エンティティ化しやすさを確認する
- テーブル責務が Java のエンティティ責務と大きくずれていないか確認する。
- 主キー、外部キー、関連の張り方が素直か確認する。

5. 論点を残す
- 命名、責務分割、制約、削除方針、更新ルールで迷う点を明示する。
- 決め切れない点は、保留ではなく「何を決めれば確定できるか」まで書く。

## Project Rules

### 命名

- テーブル名は `単数形 + snake_case` を使う。
- マスタ系テーブルは `m_*` を使う。
- トランザクション系テーブルは `t_*` を使う。
- カラム名は `snake_case` を使う。
- 主キーは原則 `*_id` を使う。
- `*_cd` は業務コードに使う。
- `*_no` は管理番号や明細番号に使う。
- 外部キーは参照先主キー名に合わせる。

### 型

- PostgreSQL 固有最適化より、Java エンティティへ起こしやすい汎用型を優先する。
- 金額は `numeric(19,4)` を使う。
- 日付は `date` を使う。
- 日時は `timestamp with time zone` を使う。
- メモや長文は `text` を使う。

### 共通カラム

- 全テーブルに `created_at` を持たせる。
- 全テーブルに `updated_at` を持たせる。

### 論理名とコメント

- テーブル名に日本語の論理名を付ける。
- カラム名に日本語の論理名を付ける。
- テーブルコメントを日本語で付ける。
- カラムコメントを日本語で付ける。
- DDL を作る場合、テーブルコメントは `COMMENT ON TABLE` で付ける。
- DDL を作る場合、カラムコメントは `COMMENT ON COLUMN` で付ける。
- `COMMENT ON` で付けるコメントは日本語を使う。

### NULL 可否

- 原則 `NOT NULL` を使う。
- 未設定が業務上意味を持つ項目だけ `NULL` を許可する。
- 主キー、必須外部キー、名称、種別、金額、日付、状態、`created_at`、`updated_at` は原則 `NOT NULL` とする。

### 削除方針

- 論理削除カラムを全表へ機械的に追加しない。
- マスタ系は `is_active` や有効期間で扱う案を優先する。
- 取引系は原則削除しない。
- 訂正が必要な場合は、取消や修正取引で表現できるかを先に検討する。

## Output

出力は原則として次を含める。

- テーブル一覧
- 各テーブルの論理名と物理名
- カラム定義
- 型
- NULL 可否
- 主キー
- 外部キー
- 一意制約
- テーブルコメント
- カラムコメント
- 設計上の論点や注意点

表形式の基本形は [references/output-format.md](references/output-format.md) を参照する。

## Notes

- この Skill は現時点では論理設計専用とし、DDL 実装や Flyway 運用は含めない。
- 方針を追加変更するときは、都度ユーザー確認を取ってから更新する。
