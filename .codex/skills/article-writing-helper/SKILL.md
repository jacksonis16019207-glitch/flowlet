# Article Writing Helper

## Overview

この Skill は、`flowlet` の開発内容を記事候補へ落とし込み、そのまま記事ドラフトまで作るときに使う。
「何を記事にするか分からない」状態から始めても、候補提案から本文作成まで順に進める。

## Workflow

1. 開発状況を確認する
- 直近で作った機能、設計判断、詰まった点、学んだこと、AI に任せたことを確認する。
- 差分、ドキュメント、コミット、会話履歴から記事化できる材料を拾う。

2. 記事候補を提案する
- いきなり本文を書かず、必ず先に記事候補を複数提案する。
- 各候補には少なくとも次を含める。
  - テーマ名
  - 想定読者
  - 何が学べるか
  - 公開先候補
  - 今書きやすい理由
- 候補が 1 つしかなさそうでも、切り口を変えて 2 件以上の候補を出すことを優先する。

3. テーマを絞る
- ユーザーに候補を選んでもらう。
- 未確定なら、「最短で書けるもの」「ポートフォリオとして説明しやすいもの」「学習価値が高いもの」を優先して推す。

4. 記事方針を決める
- 選ばれたテーマに対して、タイトル案、読者、ゴール、記事種別を整理する。
- 記事種別は主に次から選ぶ。
  - 開発記録
  - 設計解説
  - ハマりどころ共有
  - AI 活用ノウハウ
- 必要なら [references/article-patterns.md](references/article-patterns.md) を参照する。

5. アウトラインを作る
- 先に見出し構成を作る。
- 見出しは、導入、背景、やったこと、判断理由、結果、学び、次の課題を基本に組む。
- 実装の羅列ではなく、読者が追いやすい順序へ並べ替える。

6. 本文ドラフトを作る
- アウトライン確定後に本文を書く。
- 事実、判断理由、得られた学びを分けて書く。
- コード断片や設定例を入れる場合は、読者が理解に必要な最小限へ絞る。
- 不確かな内容は断定せず、「今回の判断」「現時点の理解」として書く。

7. 投稿前チェックを行う
- タイトルで内容が伝わるか確認する。
- 導入で「誰向けの記事か」が分かるか確認する。
- 途中で文脈が飛んでいないか確認する。
- 公開先に合わせて文体や見出し粒度を調整する。

## Project Rules

### 候補提案を省略しない

- 最低限の流れとして、必ず「記事候補の提案 -> 記事作成」を含める。
- ユーザーが最初から 1 テーマを指定しても、近い派生テーマを 1 件以上補足提案する。

### flowlet 向けの優先テーマ

- ポートフォリオとして説明しやすい内容を優先する。
- React / TypeScript 学習、DDD 学習、AI コーディングエージェント活用は優先的に候補へ含める。
- 単なる作業日報ではなく、判断理由や学びがある切り口を優先する。

### 記事候補の切り口

- 機能実装の流れ
- 設計の考え方
- 技術選定の理由
- バグ調査と修正
- 開発環境構築
- AI と Skills の使い分け
- 開発を通じた学習メモ

### 公開先

- 日本語中心なら `Zenn` と `Qiita` を候補にする。
- ポートフォリオ文脈や開発ストーリーを残したい場合は GitHub リポジトリ内 `docs/` やブログ下書きも候補にする。
- 公開先は 1 つに固定せず、記事の目的に合わせて提案する。

## Output

出力は原則として次を含める。

- 記事候補一覧
- 推し候補と理由
- 選ばれたテーマのタイトル案
- 想定読者と記事ゴール
- 見出しアウトライン
- 本文ドラフト
- 必要なら公開先候補と投稿時の注意点

## Notes

- 候補提案では、今の開発状況から書ける内容を優先する。
- 本文は完成品に寄せつつ、必要なら「たたき台」であることを明示する。
- 記事化しやすい区切りを見つけたら、実装タスクとは別に候補として提案する。
