# DB マイグレーション運用ルール

## 目的

`flowlet` の DB 変更で、コミット済み migration の改変、開発用 seed の混在、テーブル変更時のデータ退避漏れを防ぐ。

## 基本ルール

- Flyway はスキーマ定義と構造変更だけに使う。
- コミット済みの migration SQL は編集しない。
- DB 変更時は、既存 migration を直さずに新しい version の migration を追加する。
- 1 migration 1 関心事を基本にする。

## テーブル変更ルール

- テーブルへカラムを追加する場合は、原則として対象テーブルを再作成する。
- カラム追加以外のテーブル変更は、影響範囲、移行難度、制約変更有無を見て、再作成か局所変更かを判断する。
- 判断理由は migration コメントか関連ドキュメントに残す。

## データ保持ルール

- テーブル変更時に既存データを保持する必要がある場合は、先にバックアップスキーマを作成する。
- バックアップスキーマ名は `flowlet_backup_v<version>` を基本とする。
- 変更対象テーブルは、バックアップスキーマへデータごとコピーする。
- その後にテーブル変更を行い、新テーブル定義に合わせてバックアップデータを移行する。
- 同一 migration の中ではバックアップスキーマを削除しない。

## seed SQL ルール

- ダミーデータの投入は migration で行わない。
- 開発環境のダミーデータは `infra/sql/dev-seed/` に置く。
- 本番環境でも個人差がなく共通利用できる固定データは `infra/sql/master-data/` に置く。
- seed SQL は 1 テーブル 1 ファイルで管理する。
- seed SQL は手動実行前提とし、適用順が分かるファイル名にする。
- 再実行の可能性がある SQL は、重複投入を避ける。

## 追加ルール

- migration 名は内容が追える名前にする。
- 構造変更に合わせて `docs/setup.md`、関連仕様メモ、Skill を必要に応じて更新する。
- 既存 DB の再作成が必要な変更なら、その条件と理由を手順へ明記する。

## 現在の配置ルール

- Flyway migration: `backend/flowlet/src/main/resources/db/migration/`
- 開発用ダミーデータ: `infra/sql/dev-seed/`
- 共通マスタデータ: `infra/sql/master-data/`
