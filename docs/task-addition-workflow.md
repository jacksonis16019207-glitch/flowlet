# flowlet 今後タスク追加ワークフロー

## 目的

- 思いつきのタスク追加で優先順位が崩れないようにする
- 追加したタスクが `flowlet` の目的にどう効くかを明確にする
- 実行タスクリストを継続的に更新できるようにする

## 追加判断の原則

- `Start.md` の目的に直接効くものを優先する
- 先にある実装を壊さず、小さく検証できる単位へ分ける
- 学習目的だけのタスクは、プロダクト価値か再利用価値がある場合に追加する
- すぐ着手しないアイデアでも、記録してから優先度判断する
- CSV 取込のような現在スコープ外の項目は、必要性が再発したときだけ再評価する

## タスク追加の流れ

### 1. 追加候補を一文で書く

最初に次の形で書く。

`何をできるようにするタスクか`

例:
- ダッシュボードで未配分残高を見えるようにする
- クレジットカードの引き落とし予定を登録できるようにする

### 2. 追加理由を確認する

次の 4 点を短く埋める。

- どの課題を解くか
- どのユーースケースに効くか
- 既存のどの機能に依存するか
- 実装後に何を確認できるようになるか

この時点で答えられない場合は、タスクではなく調査メモとして扱う。

### 3. 種別を決める

追加候補を次のどれかに分類する。

- `feature`: ユーザー価値が増える機能追加
- `improvement`: 既存機能の使いやすさ、保守性、性能改善
- `bugfix`: 既存挙動の修正
- `doc`: 仕様、運用、手順の整理
- `research`: まだ実装判断前の調査

### 4. 優先度を判定する

優先度は次の基準で決める。

- `P1`: プロダクトの主要価値に直結し、今の作業を前へ進める
- `P2`: 主要価値を支えるが、直近でなくても成立する
- `P3`: あったほうがよいが、後回しでも影響が小さい
- `Backlog`: 具体化前の保留アイデア

判断に迷ったら次の順で見る。

1. 目的別残高の見える化に効くか
2. 現在残高の再現に効くか
3. 引き落とし不足の事前確認に効くか
4. 今の実装を進めるための土台になるか

### 5. 実行可能な単位へ分割する

1 タスクが大きい場合は、次の単位へ分ける。

- 設計
- backend
- frontend
- test
- docs

1 回の着手で終わらないものは、最低でも「完了条件が言える単位」まで分ける。

### 6. 記録先を決める

- すぐ着手するもの:
  [execution-task-list.md](/C:/Users/jacks/Documents/flowlet/docs/execution-task-list.md) に追加する
- 仕様や背景の整理が必要なもの:
  対応する設計ドキュメントに追記する
- 保留アイデア:
  Backlog として [execution-task-list.md](/C:/Users/jacks/Documents/flowlet/docs/execution-task-list.md) か関連ドキュメントへ短く残す

### 7. 着手前に確認する

着手前に次を確認する。

- 既存の優先順位を逆転させる必要があるか
- 既存ドキュメント更新が必要か
- テスト追加が必要か
- 記事化できる論点か
- 新しい Skill 化の価値があるか

## 更新ルール

- タスク追加時は、必ず優先度と完了条件をセットで書く
- 完了したタスクは削除せず、別ドキュメントへ移すか履歴が残る形で管理する
- 仕様変更を伴うタスクは、実装前後で関連ドキュメントを更新する
- 同じ話題が 2 回以上出たら、単なる思いつきではなく正式タスク候補として扱う

## テンプレート

以下をそのまま使う。

```md
### タスク名

種別:
- feature / improvement / bugfix / doc / research

優先度:
- P1 / P2 / P3 / Backlog

目的:
- このタスクで解決したいこと

依存:
- 先に必要な実装や設計

実行タスク:
- 実装や調査の具体項目

完了条件:
- 完了したと言える状態

メモ:
- 保留論点、記事化候補、Skill 化候補など
```
