# flowlet 実行タスクリスト

## 目的

- 今の実装状況を前提に、次に着手すべきタスクを迷わず選べるようにする
- `Start.md` の目的である「目的別残高の見える化」「現在残高の再現」「引き落とし不足の事前確認」を優先する
- 実装、検証、記事化の区切りを作りやすくする

## 前提

- `Account`、`GoalBucket`、`Category / Subcategory`、`Transaction / Transfer / GoalBucketAllocation` の基本 CRUD と画面は着手済み
- CSV 取込は現時点のスコープに含めない
- 今後の優先順位は「プロダクト価値」「学習価値」「既存実装との接続しやすさ」で決める

## 優先順位

### P1. 残高集計を完成させる

目的:
実口座残高、GoalBucket 残高、未配分残高を一貫したルールで確認できる状態にする

実行タスク:
- ダッシュボードで必要な表示項目を確定する
- 残高集計 API のレスポンス案を決める
- `Account`、`GoalBucket`、未配分残高の計算ロジックを整理する
- バックエンドに残高集計 API を追加する
- フロントエンドにダッシュボード画面を追加する
- 残高計算ルールのテストを追加する

完了条件:
- 口座ごとの現在残高を画面で確認できる
- GoalBucket ごとの残高を画面で確認できる
- 未配分残高を画面で確認できる
- 取引、振替、配分を行った結果が集計へ反映される

### P2. 取引系のテストを補強する

目的:
既に入っている取引系機能を安全に拡張できる状態にする

実行タスク:
- `Transaction` の正常系と異常系を整理する
- `Transfer` の 2 明細生成をテストする
- `GoalBucketAllocation` の複数配分展開をテストする
- 削除時の関連データ削除をテストする
- 残高計算に影響する主要ケースをテストへ追加する

完了条件:
- 取引、振替、配分の主要ルールが自動テストで固定されている
- 回帰時に壊れた箇所をテストで検知できる

### P3. 開発用 seed データを整理する

目的:
画面確認、手動検証、記事用スクリーンショットの準備を速くする

実行タスク:
- 公開用に使えるダミーデータ方針を再確認する
- 固定マスタデータを `infra/sql/master-data/` に整理する
- 開発用データを `infra/sql/dev-seed/` に整理する
- 取引、振替、配分、残高確認がすぐできるデータセットを作る
- seed 投入手順を `docs/setup.md` に追記する

完了条件:
- ローカル環境で seed を投入すれば主要画面をすぐ確認できる
- 公開してよいデータだけで動作確認できる

### P4. 月次収支を集計・表示できるようにする

目的:
残高の再現に加えて、一定期間の収入、支出、差額を確認できるようにする

実行タスク:
- 月次収支で確認したい単位を整理する
- 収支集計に含める取引種別と除外ルールを決める
- 振替や配分を収支に含めるかどうかを整理する
- 月次収支の集計 API を設計する
- ダッシュボードまたは専用画面で月次収支を表示できるようにする
- 集計ルールのテストを追加する

完了条件:
- 一定期間の収入、支出、差額を確認できる
- 振替など収支に含めない取引の扱いが明確になっている
- 月次収支の初版を画面で確認できる

### P5. ScheduledPayment を実装する

目的:
引き落とし不足の事前確認という主要ユースケースを成立させる

実行タスク:
- `ScheduledPayment` の責務を定義する
- DB テーブルと migration を追加する
- CRUD API を追加する
- 不足判定ロジックを追加する
- 一覧画面と登録フォームを追加する
- 不足見込み表示をダッシュボードまたは専用画面へ追加する
- 業務ルールのテストを追加する

完了条件:
- 引き落とし予定を登録できる
- 対象口座ごとの予定額合計を確認できる
- 不足見込みの有無を確認できる

### P6. UI の整理と操作性改善

目的:
今ある機能を日常的に使いやすい形へ整える

実行タスク:
- 取引画面の情報量を整理する
- 登録フォームと一覧の責務を分離する
- 取引画面で必要な情報だけを先に見せ、詳細確認の導線を分ける
- 取引登録中に口座を追加できる導線を検討する
- クレジットカード口座の入力項目を見直し、提供元名の扱いを整理する
- クレジットカードの締め日に月末を指定できるようにする
- 日付入力をキーボードでも行いやすくし、手入力とカレンダー選択を両立する
- 金額入力時に四則演算を使えるようにし、手計算をその場で反映できるようにする
- よく使う取引を少ない操作で登録できるようにし、定型入力を簡略化する
- フィルタ、並び順、表示グルーピングの必要性を確認する
- ダッシュボードから各画面への導線を整理する
- エラーメッセージと空状態表示を見直す
- モバイル幅での崩れを確認する

完了条件:
- 主要画面で次に押す操作が分かりやすい
- 一覧と登録フォームの往復がしやすい
- PC とスマホ幅の両方で最低限の操作が成立する

### P7. ローカル本番環境のモバイル確認手段を整備する

目的:
PWA 対応前でも、スマホやタブレットからローカル本番環境を安全に確認できる状態にする

実行タスク:
- SSH port forward を前提にした接続手順を整理する
- iPhone / iPad / Android / タブレットでの確認手順を必要最小限で文書化する
- ローカル本番環境でモバイル表示確認に必要な前提条件を洗い出す
- PWA 対応後はこの暫定手段を卒業する前提をドキュメントに残す

完了条件:
- 手元端末からローカル本番環境へ接続して主要画面を確認できる
- 接続手順と前提条件が `docs/` に残っている
- この対応が暫定運用であることを後から判断できる

### P8. 検索と絞り込みを追加する

目的:
既存の主要機能が揃った後に、口座、取引、カテゴリを探しやすくする

実行タスク:
- 口座、取引、カテゴリごとに必要な検索条件を整理する
- 前方一致、部分一致、状態絞り込みなどの最小仕様を決める
- 主要な一覧画面で必要なソート条件と昇順・降順の切り替え仕様を決める
- API 側と UI 側の責務を整理する
- 既存一覧画面へ無理なく組み込める導線を検討する

完了条件:
- 口座、取引、カテゴリを最低限の条件で検索できる
- 主要な一覧画面で必要なソートを使える
- 一覧画面の操作性を大きく崩さずに検索を使える

### P9. PWA と公開準備に着手する

目的:
ローカル中心の利用から、モバイルで使いやすい配布形態へ段階的に移行する

実行タスク:
- PWA 化の対象範囲を整理する
- manifest、icon、installable 条件、更新方針を決める
- PWA 対応後のアクセス方式として SSH 前提を外す
- 独自ドメイン取得と HTTPS 化の前提条件を整理する
- PWA 後に必要となる `User`、認証、認可の導入順を整理する

完了条件:
- PWA 対応の着手条件と設計論点が整理されている
- 公開準備に必要な後続タスクが順序付きで見える
- SSH 接続から卒業する前提が整理されている
