# flowlet の要件定義: 目的別残高の見える化と引き落とし不足判定をどう整理したか

## はじめに

`flowlet` は、実在する銀行口座とアプリ内の目的別口座を分けて管理するための Web アプリです。

作り始める前に整理したかったのは、単に「家計管理アプリを作る」という大きな話ではなく、自分が今どこに困っていて、その困りごとをどこまで MVP で解くのかという点でした。

この記事では、`flowlet` の要件定義で特に重視した 2 つの論点を整理します。

- 目的別残高の見える化
- 引き落とし不足判定

## 先に結論

今回の要件定義では、次のように考えました。

1. 実口座の残高管理と、目的別の資金配分管理は分けて考える
2. 家計簿の分析機能より先に、今の残高と近い将来の不足確認を優先する
3. MVP では、記録と判定に必要な最小限の入力に絞る

この切り方にしたことで、最初から何でも入れようとして要件がぼやけるのを避けられました。

## 背景: 何を解きたかったのか

もともと困っていたのは、収支分析そのものではありませんでした。

実際に困っていたのは、次の 2 点です。

- 1 つの実口座にまとまった資金を、用途別に把握しづらい
- カードや固定費の引き落とし予定に対して、残高が足りるかを毎回手計算している

この時点で、一般的な家計簿アプリでよくある「カテゴリごとの支出分析」や「月次レポート」は主目的ではないと分かりました。

`flowlet` では、まず「今あるお金をどう置いているか」と「近い将来に足りるか」を扱うべきだと判断しました。

## 論点1: 目的別残高の見える化

最初の論点は、実口座と目的別口座をどう分けて考えるかでした。

銀行側の機能で目的別管理ができていたときは、用途ごとの残高をそのまま見られました。  
しかし、資金を集約すると、実際の残高は見えても、その中身の意味が分かりにくくなります。

そこで `flowlet` では、次のように整理しました。

- `Account`
  - 実在する銀行口座
- `GoalBucket`
  - アプリ内で管理する目的別口座

この分け方にすると、現実の口座構成をそのまま保持しつつ、アプリ上で「生活イベント」「定期支払」「投資用」といった目的単位を再構成できます。

ここで大事だったのは、実口座を増やして問題を解こうとしなかったことです。  
現実の運用を無理に変えるのではなく、アプリ側のモデルで解決する方が、運用にも実装にも筋がよいと考えました。

## 論点2: 引き落とし不足判定

もう 1 つの論点は、引き落とし予定をどう扱うかです。

実際の困りごとは、「支払い履歴を後で分析したい」というより、「次の引き落とし日に足りるかを事前に知りたい」でした。

そのため、MVP では次の情報を手入力で持てれば十分だと整理しました。

- 支払い名
- 金額
- 引き落とし日
- 対象口座
- 繰り返し有無

この情報があれば、対象口座ごとに今後の引き落とし予定額を集計し、現在残高と比較して不足の有無を判定できます。

つまり、最初から銀行 API 連携や自動取り込みをやらなくても、価値の中心である「不足の事前確認」は実現できます。

## なぜ 2 つを分けて考えたのか

要件定義の段階では、「目的別残高」と「引き落とし不足判定」をひとまとめに見てしまいがちでした。

ただ、実際には役割が違います。

- 目的別残高の見える化
  - 今ある資金を意味ごとに把握するためのもの
- 引き落とし不足判定
  - 近い将来の支払いに耐えられるかを確認するためのもの

この 2 つを分けておくと、モデルも UI も考えやすくなります。

たとえば、目的別残高は「用途ごとの配置」を見せる画面に向いています。  
一方で引き落とし不足判定は、「どの口座で、いつ、いくら足りないか」を見せる画面に向いています。

似ているようで、読者に見せたい情報の軸が違います。

## MVP に入れるものと入れないもの

MVP では、次を Must にしました。

- 実口座の登録
- 目的別口座の登録
- 取引の手入力
- 目的別残高の表示
- 未配分残高の表示
- 引き落とし予定の手入力
- 対象口座の引き落とし不足判定
- CSV 取込の初版

逆に、最初は入れないものも明確にしました。

- 銀行 API 連携
- 自動同期
- 複数人利用
- 本格的な家計簿分析機能

ここを切れたのは大きかったです。  
要件定義の段階で「便利そうだから入れる」を抑えられたので、最初の実装対象がかなり見えやすくなりました。

## 要件定義で意識したこと

今回の要件定義では、一般論よりも、自分の運用に対して本当に必要なものを優先しました。

意識したことは次の 3 つです。

1. 困りごとをそのまま言葉にする
2. 画面やテーブルより先に、何を判断したいかを決める
3. 学習用アプリでも、現実の運用に耐える軸を持たせる

個人開発だと、技術選定や実装の方に意識が向きやすいです。  
ただ、今回は最初に「何を確認できるようになれば価値があるか」をはっきりさせたことで、その後の設計と実装がかなり進めやすくなりました。

## おわりに

`flowlet` の要件定義では、「実口座と目的別口座を分けること」と「引き落とし不足を事前確認できること」を中心に据えました。

この 2 つを先に整理したことで、家計簿アプリとして機能を広げる前に、まずは自分に必要な価値を MVP として切り出せたと思っています。

次は、この要件をもとに、最初の縦切り実装として口座マスタ管理をどう作ったかをまとめる予定です。
