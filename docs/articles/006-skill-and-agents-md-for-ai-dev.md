# flowlet で学んだ、AI コーディング運用を Skill と AGENTS.md に落とし込むやり方

## はじめに

AI コーディングエージェントを使った個人開発では、実装そのものだけでなく「どう進めるか」のルールもかなり重要です。
最初のうちは会話で都度伝えれば回るのですが、タスク数が増えると同じ指示を何度も繰り返すことになります。

`flowlet` の開発でも、まさにその状態が起きました。
そこで今は、繰り返し出てくる運用ルールを Skill と `AGENTS.md` に分けて残すようにしています。

この記事では、`flowlet` で実際にやったことをもとに、AI コーディング運用のルールをどのように整理しているかを書きます。

## なぜルールを毎回会話で伝えるだけでは足りなくなったか

AI エージェントと一緒に開発していると、最初は「この方針で進めてください」とその場で伝えるだけでも進みます。
ただ、何タスクか進めるうちに、次のようなルールが何度も出てくるようになりました。

- 変更前には短く提案して承認を取る
- 最小変更を優先する
- ドキュメント更新が必要なら忘れずに行う
- コミットは Conventional Commits 形式で残す
- 作業フローは `task-delivery-flow` に沿って進める

こうしたルールを毎回会話で補うのは、地味にコストが高いです。
しかも、同じルールでも表現が揺れると、エージェント側の振る舞いも少しずつぶれます。

そのため、繰り返し出るルールは会話から外に出して、参照できる形にしておいたほうが安定すると感じました。

## flowlet で使っている 2 つの仕組み

`flowlet` では、主に Skill と `AGENTS.md` を使っています。

Skill は、特定の作業をどう進めるかをまとめた手順書です。
たとえば `task-delivery-flow` には、要件確認、設計確認、実装、検証、ドキュメント更新、コミットまでの流れをまとめています。

一方で `AGENTS.md` は、リポジトリ全体で守りたい運用ルールを書く場所です。
言語、承認の取り方、編集方針、Git の扱い、ドキュメント更新の考え方など、広い範囲に効かせたい内容をここに置いています。

自分の中では、次のように分けると整理しやすいです。

- Skill: タスクの進め方
- `AGENTS.md`: リポジトリで守る共通ルール

## Skill に入れるもの、AGENTS.md に入れるもの

この 2 つは似て見えるのですが、役割は少し違います。

Skill に向いているのは、順番や判断ステップがあるものです。
たとえば「実装前に何を確認するか」「実装後にどこまで検証するか」「記事化や Skill 改善提案をいつ検討するか」といった内容です。

逆に `AGENTS.md` に向いているのは、毎回の作業で前提になるルールです。
たとえば「日本語で応答する」「変更前に承認を得る」「最小変更を優先する」「コミットは行う」といった内容です。

今回あらためて分かったのは、手順と規約を同じ場所に詰め込まないほうがよいということでした。
手順は Skill、規約は `AGENTS.md` と分けたほうが、あとから見返したときにも迷いません。

## 今回の開発で実際にルール化した内容

最近の `flowlet` 開発では、2 つのことを追加でルール化しました。

1 つ目は、`task-delivery-flow` に「AGENTS.md の改善提案を検討する」流れを加えたことです。
開発を進めていると、「このルールは次回以降も必要そうだな」と気づく場面があります。
でも、その気づきを毎回そのまま流してしまうと、結局また同じ話を会話で繰り返すことになります。

そこで、タスク完了前に「AGENTS.md に足すべき運用ルールがあるか」を確認するステップを入れました。
不要ならそのままスルーしてよい、という形にしているのもポイントです。

2 つ目は、文字コード方針を `AGENTS.md` に追加したことです。
実際にリポジトリを確認すると、UTF-8 BOM 付きや Shift_JIS のファイルが混ざっていました。
その場で直すだけでもよかったのですが、再発防止まで考えると「今後は UTF-8 BOMなしで統一する」と明文化したほうが運用しやすいと判断しました。

さらに `.editorconfig` にも `charset = utf-8` を追加して、会話ルールだけでなくツール側でも寄せるようにしました。

## ルールを増やすときに意識していること

ただし、思いつくたびに全部ルール化すればよいわけではありません。
やりすぎると、読む量だけが増えて逆に使いづらくなります。

自分が意識しているのは次の 3 点です。

- 同じ指示を 2 回以上出したか
- 今後も再利用されそうか
- Skill と `AGENTS.md` のどちらに置くと自然か

たとえば、その場限りの事情なら会話で済ませたほうが速いです。
一方で、今後も高い確率で繰り返す内容なら、どこかに残したほうが結果的に楽になります。

もう 1 つ大事なのは、ルールを増やすなら「なぜ必要か」を自分で説明できる状態にしておくことです。
理由が弱いルールは、あとで自分でも守りにくくなります。

## まとめ

AI コーディングエージェントを使った個人開発では、コードだけでなく運用ルールの置き場を整えることも重要だと感じています。

`flowlet` では、作業の進め方は Skill に、リポジトリ全体の共通ルールは `AGENTS.md` に分けることで、会話のたびに同じ指示を繰り返す量を減らしてきました。

特に、開発の途中で見つかった運用上の気づきを「その場の会話」で終わらせず、再利用できる形に変えていく流れはかなり効いています。

AI にコードを書いてもらうなら、実装ルールも少しずつ外だししていく。
それが、個人開発を安定して続けるためのひとつのコツだと思っています。
