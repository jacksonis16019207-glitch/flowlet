# SBI の目的別口座を失って困ったので、家計・資産管理アプリを作ることにした

## はじめに

家計管理や資産管理のアプリはすでにたくさんあります。

それでも自分で `flowlet` という Web アプリを作ることにしたのは、既存のアプリではなく、今の自分の口座運用そのものに合った仕組みが欲しかったからです。

この記事では、何に困っていたのか、なぜ作ることにしたのか、そして `flowlet` で何を解決したいのかを整理します。

## 何に困っていたのか

もともと SBIネット銀行では、目的別口座を使ってお金を用途ごとに分けて管理していました。

たとえば、次のような単位です。

- 親孝行
- 定期支払
- 特別貯金
- 衣服・美容

さらに、投資用のお金はハイブリッド預金で分けていました。

ところが、SBIネット銀行の口座を解約し、SBI新生銀行に資金を移したことで状況が変わりました。  
SBI新生銀行では `普通預金` と `ハイパー預金` を使っていますが、ハイパー預金は目的別に分かれていません。

つまり、実際のお金は 1 つの預金にまとまっているのに、その中で

- 何が親孝行用なのか
- 何が定期支払用なのか
- 何が投資用なのか

が見えにくくなりました。

これまでは銀行の機能で自然にできていたことを、自分の頭やメモで補う必要が出てきたわけです。

## もう1つの困りごと

困っていたのは、目的別残高の見えにくさだけではありません。

三菱UFJ銀行のメイン口座では、給与受取や各種カードの引き落としを管理しています。  
そのため、引き落とし日が近づくたびに、「今の残高で足りるか」を計算機で確認する必要がありました。

これは地味ですが、かなり面倒です。

- どのカードがいくら引かれるかを確認する
- 合計額を計算する
- 今の口座残高と見比べる
- 足りないならどこから移すか考える

この作業は毎回似ているのに、仕組みとしては残っていません。  
都度手で確認しているので、抜け漏れも起きやすくなります。

## 作りたいのは家計簿よりも資金配置アプリ

今回作りたいものは、一般的な家計簿アプリをそのまま目指すものではありません。

もちろん収支の記録は必要です。  
ただ、それは家計分析のためというより、今の残高を正しく再現するために必要です。

自分にとって本当に必要なのは、次の 2 つでした。

1. 実在する銀行口座の残高を把握できること
2. その残高を、アプリ上では目的別に分けて見えること

つまり、銀行の実口座とアプリ内の目的別口座を分けて扱える仕組みが必要でした。

たとえば、SBI新生銀行の `ハイパー預金` が 1 口座しかなくても、アプリ上では

- 親孝行
- 定期支払
- 特別貯金
- 衣服・美容
- 投資用

のように、仮想的な口座として分けて持てるようにしたいと考えています。

## flowlet で解決したいこと

`flowlet` では、まず次のことをできるようにしたいです。

- 実口座を登録する
- 目的別口座を作る
- 入出金や振替を記録する
- 実口座残高と目的別残高を見える化する
- 引き落とし予定を登録する
- 対象口座の残高が引き落とし予定額に足りるかを確認する

最初の MVP では、特に次の価値を重視しています。

- `SBI新生銀行` の中で失われた目的別管理を、アプリ側で復元すること
- `三菱UFJ銀行` のメイン口座で、引き落とし不足を事前に確認できるようにすること

## なぜ今自分で作るのか

理由は 2 つあります。

1 つは、今の自分の不便に対して最もまっすぐ効くものを作れるからです。  
汎用的な家計簿アプリは多いですが、自分の銀行運用や目的別管理の癖にぴったり合うものは限られます。

もう 1 つは、学習の軸としてちょうどよい題材だからです。

`flowlet` は次の学習目的も兼ねています。

- React / TypeScript を使ったフロントエンド開発
- Java / Spring Boot を使ったバックエンド開発
- DDD を意識したドメイン整理
- AI コーディングエージェントを活用した個人開発

単なる練習用アプリではなく、自分の生活上の問題を解決しながら、技術的にも積み上がる題材にしたいと思っています。

## 技術スタック

現時点では、次の構成で進めています。

- backend: Java 25 + Spring Boot 4
- frontend: React 19 + Vite 8 + TypeScript
- database: PostgreSQL 18
- infra: Docker

この構成にしたのは、モダンな Web アプリとしての作りやすさと、今後の学習価値の両方を取りたかったからです。

## おわりに

`flowlet` は、家計をきれいに分析するためのアプリというより、今あるお金をどこにどう置いているのかを崩さず管理するためのアプリとして始めます。

まずは、目的別残高の見える化と引き落とし不足判定を MVP として形にしていく予定です。

次の記事では、`flowlet` の要件定義をどう整理したか、特に「目的別残高」と「引き落とし不足判定」をどのように分けて考えたかを書いていきます。
