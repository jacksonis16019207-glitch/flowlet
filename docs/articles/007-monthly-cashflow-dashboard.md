# 残高集計の次に、月次収支をどう切り出したか: flowlet でやったこと

## はじめに

個人開発で家計・資産管理アプリを作っていると、「残高が見える」と「収支が見える」は似ているようで、実は少し違う論点だと感じます。

`flowlet` では、先に「実口座残高」「GoalBucket 残高」「未配分残高」を確認できるダッシュボードを作りました。  
その次の段階として追加したのが、月次収支の集計と表示です。

この記事では、なぜ残高集計の次に月次収支を入れたのか、どの取引を収支に含めて何を除外したのか、実装時にどう切り出したのかを整理します。

## 先に残高集計を作っていた理由

`flowlet` の主要な課題は、まず次の 3 つです。

- 目的別残高の見える化
- 現在残高の再現
- 引き落とし不足の事前確認

この中で、最初に強く必要だったのは「今いくらあるか」を把握することでした。

そのため、先にダッシュボードで次を確認できる状態を作っています。

- 実口座ごとの現在残高
- GoalBucket ごとの現在残高
- 口座ごとの未配分残高

ここが整っていない段階で月次収支を入れると、アプリの中心が「家計簿的な分析」に寄りすぎる気がしていました。  
`flowlet` は家計簿アプリというより、まず「資金配置の見える化」を優先したかったので、順番としては残高集計が先でした。

## それでも月次収支を次に入れた理由

残高だけ見えても、「今月どれだけ増えたか、減ったか」は分かりません。

残高は結果ですが、収支は変化の流れです。  
今後 `ScheduledPayment` による不足見込みまで進めるためにも、期間単位の見え方を先に持っておく価値があると考えました。

また、月次収支は次の点でもちょうどよい拡張でした。

- 既存の `Transaction` を再利用できる
- 新しい書き込み機能を増やさず、読み取り API だけで追加できる
- ダッシュボードの文脈のまま自然に表示を足せる
- 「分析機能を広げすぎない初版」を作りやすい

つまり、新しい概念を増やしすぎずに、ダッシュボードを一段前に進められる題材でした。

## どこまでを月次収支に含めるか

今回いちばん整理が必要だったのは、どの取引を「収支」とみなすかです。

`flowlet` には通常の収入・支出だけでなく、次のような動きがあります。

- 口座間の振替
- `GoalBucket` への配分
- `GoalBucket` をひも付けた取引

ここを曖昧にすると、月次収支が実態より大きく見えたり、逆に意味がぶれたりします。

初版では、次のルールにしました。

- `transactionType = INCOME` は収入に含める
- `transactionType = EXPENSE` は支出に含める
- `TRANSFER_IN` / `TRANSFER_OUT` は収支に含めない
- `GoalBucketAllocation` は収支に含めない
- `goalBucketId` 付きの `INCOME` / `EXPENSE` は収支に含める

理由は単純で、「実際の収入・支出かどうか」を優先したかったからです。

振替や配分は資金の置き場所を変えるだけで、家計全体としての増減ではありません。  
逆に、`GoalBucket` をひも付けていても、それが実際の支出や収入なら月次収支には入れたいと考えました。

## API は残高集計と分けた

実装上は、月次収支を既存の残高集計 API に混ぜず、別 API にしました。

- `GET /api/dashboard/balance-summary`
- `GET /api/dashboard/monthly-cashflow`

こうしたのは、責務を分けたかったからです。

残高集計 API は「今の状態」を返します。  
月次収支 API は「期間ごとの変化」を返します。

同じダッシュボード用途ではありますが、返すデータの粒度も、絞り込み条件も、今後の拡張方向も違います。  
最初から分けておく方が、後でカテゴリ別収支や前月比を足すときにも扱いやすいと判断しました。

## 実装でやったこと

バックエンドでは、月次収支専用の読み取りサービスを追加しました。

- `DashboardMonthlyCashflowService`
- `DashboardMonthlyCashflowResponse`
- `DashboardMonthlyCashflowMonthResponse`
- `DashboardMonthlyCashflowTotalsResponse`

処理の流れはかなり単純です。

1. `fromMonth` と `toMonth` を受け取る
2. 月の形式と範囲を検証する
3. 指定された月の配列を先に作る
4. 既存 `Transaction` を走査して、対象月へ `INCOME` と `EXPENSE` を加算する
5. 月ごとの `income` / `expense` / `net` と、期間合計を返す

初版では、パフォーマンス最適化よりもルールの明確さを優先して、既存 `TransactionRepository` の一覧取得から組み立てています。

フロントエンドでは、ダッシュボードに月次収支セクションを追加しました。

- 直近 4 か月を対象に取得
- 月ごとの収入、支出、差額をカード表示
- 期間全体の合計も同時に表示

まずは「見える」ことを優先し、グラフやカテゴリ別内訳は次段階へ回しています。

## テストで固定したこと

月次収支は、見た目より先にルールをテストで固定したかったので、Controller テストを追加しました。

特に押さえたのは次の点です。

- 月ごとの収入、支出、差額が正しく返ること
- 対象期間に取引がない月も `0` で返ること
- `TRANSFER_IN` / `TRANSFER_OUT` が集計に入らないこと
- `fromMonth` / `toMonth` の形式エラーを返すこと
- 逆順指定や広すぎる範囲を弾くこと

このあたりを先に固めておくと、後でカテゴリ別内訳や表示変更を入れても、集計ルールそのものは崩れにくくなります。

## 実装して見えたこと

今回やってみて分かったのは、「残高」と「収支」は似ていても、API としては別物として扱った方が整理しやすいということでした。

残高は `balanceSide` や `GoalBucketAllocation` の影響を強く受けます。  
一方で月次収支は、「その月に実際の収入・支出があったか」を基準にした方が分かりやすいです。

この切り分けが見えたことで、今後の `ScheduledPayment` に向けても整理しやすくなりました。  
不足見込みは「今の残高」と「近い将来の支払い予定」を重ねて見る機能なので、今回の月次収支追加はその中間ステップとしても意味があります。

## 次にやりたいこと

次の候補として考えているのは次のあたりです。

- `ScheduledPayment` の設計と実装
- 月次収支のカテゴリ別内訳
- 前月比の表示
- グラフ表現の追加

ただ、まずは `flowlet` の主目的である「引き落とし不足の事前確認」を成立させたいので、次は `ScheduledPayment` を優先する予定です。

## おわりに

月次収支は家計アプリでは定番の機能ですが、`flowlet` では残高集計の次に入れることで、役割をかなり整理しやすくなりました。

いきなり分析機能を広げるのではなく、今のアプリの中心にある価値とつながる形で 1 段ずつ積み上げる方が、個人開発では進めやすいと感じています。

今回の追加も、単に「表示項目を増やした」というより、残高集計の次にどの読み取り機能を入れるべきかを考えた結果でした。  
同じように、家計・資産管理系のアプリを段階的に作っている人には参考になるかもしれません。
