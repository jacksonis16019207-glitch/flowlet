# 個人開発で Codex Skills を育てながら進める: flowlet で試しているやり方

## はじめに

AI コーディングエージェントを使って個人開発を進めるとき、最初は「とりあえず何でも聞く」状態になりがちです。

それでも短期的には進みますが、だんだん毎回同じ説明をしたり、判断基準がぶれたりして、使い方が安定しません。

`flowlet` では、そうした状態を減らすために、Codex の Skills を少しずつ育てながら開発を進めています。

この記事では、なぜ Skills を使うのか、どんなものを Skill 化するとよいのか、実際にどう役立っているかを整理します。

## なぜ Skill 化したいのか

`flowlet` は単なるアプリ開発ではなく、次の学習目的も持っています。

- React / TypeScript を学ぶ
- Spring Boot を学ぶ
- DDD を意識した責務分離を学ぶ
- AI コーディングエージェントの活用を学ぶ

この中で AI 活用だけが特別なのではなく、他の学習テーマと同じくらい「どう再現性を持たせるか」が重要だと感じました。

そこで、毎回同じように考えることは Skill に寄せる方針を取りました。

## flowlet で優先している Skill

現時点では、`flowlet` では project-local Skill を優先して使うようにしています。

特に軸にしているのは次のものです。

- `db-design-helper`
- `java-spring-helper`
- `react-typescript-helper`
- `task-breakdown-helper`
- `test-design-helper`

さらに今回、記事化を進めやすくするために `article-writing-helper` も追加しました。

この構成にしたのは、実装そのものだけでなく、設計、タスク分解、検証、発信まで一連の流れを補助したかったからです。

## Skill があると何が変わるか

Skill を使う前と後で変わったのは、主に次の点です。

1. 着手時の観点がそろう
2. 出力の粒度が安定する
3. 次に何を決めるべきかが見えやすくなる

たとえば設計相談でも、毎回ゼロから聞くと、ある回では API だけ、別の回では DB だけというように視点がぶれます。  
Skill があると、「何を確認してから答えるか」がある程度固定されるので、議論の抜けが減ります。

記事作成でも同じです。  
いきなり本文を書くのではなく、候補提案から始める流れを Skill に入れておくことで、「書けそうなことがあるのに記事にならない」状態を減らせます。

## どんなものを Skill にするとよいか

個人開発で Skill 化しやすいのは、次のような仕事です。

- 毎回似た観点で判断するもの
- 出力フォーマットを揃えたいもの
- 実装そのものではなく、前後の整理が必要なもの
- 将来的に何度も繰り返すもの

逆に、1 回限りの細かい実装詳細は、無理に Skill にしなくてもよいと思っています。

`flowlet` では特に、次の領域が Skill と相性がよいと感じています。

- 要件定義
- 設計整理
- フロント実装方針
- バックエンド実装方針
- テスト観点整理
- 記事化
- Git 運用

## Skill は制約でもある

Skill は便利ですが、強すぎるテンプレートにすると逆に考えなくなります。

そのため、`flowlet` では「毎回同じ結論を出すためのもの」ではなく、「最低限見るべき観点を揃えるためのもの」として扱うのがよいと感じています。

たとえば、React 側の実装でも常に同じ構成が正しいとは限りません。  
ただ、「画面、API、型、コンポーネントの責務を意識する」といった観点は繰り返し使えます。

つまり、Skill は答えを固定するものというより、思考のガードレールに近いです。

## 記事作成 Skill を作ってよかったこと

今回 `article-writing-helper` を追加して特によかったのは、開発と発信を切り離さずに済むようになったことです。

個人開発では、記事にしたいと思っていても、実装が進むほど後回しになりがちです。  
しかも、「何を書けばよいか分からない」という状態が続くと、そのまま何も残りません。

記事作成 Skill では、次の流れを先に決めました。

1. 開発状況を確認する
2. 記事候補を複数出す
3. テーマを絞る
4. アウトラインを作る
5. 本文ドラフトを書く

この流れがあるだけで、記事作成の心理的ハードルはかなり下がります。  
今回も、要件定義、縦切り実装、公開データ方針、AI 活用という形で複数の下書きを切り出しやすくなりました。

## 今後 Skill 化したいもの

今後さらに Skill 化するとよさそうだと感じているのは次のような領域です。

- API 仕様の差分整理
- MVP ごとの受け入れ条件整理
- デモ準備チェック
- 公開記事の投稿前チェック

アプリを作るだけでなく、見せる、説明する、運用するところまで含めると、Skill にできる余地はまだかなりあります。

## おわりに

`flowlet` では、AI コーディングエージェントを単なる便利ツールとして使うのではなく、Skills を通じて再利用できる形に整えながら進めています。

個人開発ではスピードも大事ですが、それ以上に「次も同じ水準で進められるか」が重要です。  
Skill は、そのための土台としてかなり有効だと感じています。
