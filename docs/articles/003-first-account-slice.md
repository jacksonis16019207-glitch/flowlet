# Java + Spring Boot + React で最初の「口座マスタ管理」を通した話

## はじめに

新しく Web アプリを作るとき、最初にどの機能から作るかは意外と迷います。

`flowlet` でも、最終的には取引管理や引き落とし予定、目的別残高の表示までやりたいと考えています。  
ただ、最初からそこまで広げると、フロントエンドもバックエンドも設計が重くなりがちです。

そこで最初の縦切り実装として選んだのが、「口座マスタ管理」でした。

この記事では、なぜ口座マスタから始めたのか、Spring Boot と React をどうつないだのか、その段階で見えてきた学びを書きます。

## なぜ口座マスタから始めたのか

`flowlet` では、ほとんどの機能が口座情報を前提にします。

- 取引はどの口座に対するものか
- 引き落とし予定はどの口座から落ちるのか
- 目的別口座はどの実口座に属するのか

つまり、口座を表す土台がないと、その先の機能を自然に積み上げにくい状態でした。

また、口座マスタは次の点でも最初の題材として扱いやすいです。

- 業務的に意味が分かりやすい
- 登録と一覧で最小の CRUD を作れる
- API と UI を最短距離でつなげられる
- テストを書きやすい

最初の一歩として、ちょうどよいサイズでした。

## 今回作った範囲

現時点で実装しているのは、主に次の流れです。

1. 口座を登録する
2. 登録済みの口座一覧を取得する
3. フロントエンドで登録フォームと一覧画面を表示する

実装としては、バックエンドに `GET /api/accounts` と `POST /api/accounts` を用意し、フロントエンドから呼び出しています。

この時点では、編集や削除、検索などはまだ入れていません。  
まずは最小限の流れを通すことを優先しました。

## バックエンド側の考え方

Spring Boot 側では、Controller から直接永続化の詳細を触るのではなく、サービスとリポジトリを挟む形にしています。

大きな流れとしては次のとおりです。

- `AccountController`
  - API の入口
- `AccountService`
  - 作成や一覧取得のユースケース
- `AccountRepository`
  - ドメイン側の保存・取得インターフェース
- JPA 実装
  - 永続化の詳細

この構成にしたのは、今後 `GoalBucket` や `Transaction` を追加するときにも責務を揃えやすくしたかったからです。

個人開発なので厳密にレイヤを増やしすぎるつもりはありません。  
それでも、Controller にロジックを寄せすぎない形だけは最初に作っておいた方が、あとで崩れにくいと考えました。

## フロントエンド側の考え方

React 側では、画面、API 呼び出し、表示コンポーネント、型定義を分けています。

具体的には、次のような分割です。

- `pages/accounts`
  - 画面の入り口
- `features/account/api`
  - API 呼び出し
- `features/account/components`
  - フォームと一覧表示
- `features/account/types`
  - `Account` や `CreateAccountInput` の型

この分け方にしたのは、「1 画面 1 ファイル」に寄せすぎると、後で責務が混ざりやすいからです。  
最初の段階から、機能単位で整理する形にしておく方が、React / TypeScript の学習にもつながると考えました。

## 最小実装で意識したこと

最初の実装では、見た目を豪華にするより、次の 3 点を優先しました。

1. 入力から保存までが実際に通ること
2. 失敗時に最低限のエラーメッセージが出ること
3. バックエンドのテストが先に置けること

特に重要だったのは、テスト込みで API の骨格を先に固めたことです。

`POST /api/accounts` で口座を登録できること、`GET /api/accounts` で登録済みデータを返せることをテストで押さえておくと、後からフロント側を触っても不安が減ります。

最初の段階で「画面はできたけど API が曖昧」という状態にしなかったのはよかった判断でした。

## 口座マスタを作って見えたこと

小さい機能ですが、これを通したことで次の論点がはっきりしました。

- 実口座の属性はどこまで最初に持つべきか
- 目的別口座との関連をどのタイミングで入れるか
- 一覧の並び順や表示項目をどこまで要件に含めるか
- `active` の扱いを今後どう統一するか

つまり、口座マスタ自体が完成品というより、その先の設計を具体化するための足場になっています。

最初の機能を小さく切ると、実装量が減るだけでなく、次の判断材料も得やすくなります。

## 学び

今回の縦切り実装で特に学びが大きかったのは次の点です。

- 最初の機能は「全部の前提になるもの」から選ぶと進めやすい
- フロントとバックを同時に少しずつ通すと、設計のズレに早く気づける
- 個人開発でも、責務の分離を最初に少し入れておく価値は大きい

`flowlet` は学習目的も兼ねているので、React / TypeScript、Spring Boot、DDD っぽい責務分離を小さく試せるこの題材はかなり相性がよかったです。

## おわりに

最初の縦切り実装として口座マスタ管理を選んだことで、`flowlet` の土台がかなり具体的になりました。

次は、この口座情報を前提にして、目的別口座や取引、引き落とし予定へ広げていく予定です。  
その過程でも、なるべく「小さく通す」を崩さず進めたいと思っています。
