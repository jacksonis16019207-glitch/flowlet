# 個人開発アプリを公開するとき、実データをどう避けるか: flowlet のダミーデータ方針

## はじめに

個人開発アプリをポートフォリオとして見せたいと思ったとき、意外と困るのがデータの扱いです。

特に `flowlet` のように、口座情報や支払い予定のような個人性が高いデータを扱うアプリでは、実データをそのまま公開用環境に置くわけにはいきません。

一方で、データが何もない状態だと、アプリの価値も伝わりにくくなります。

この記事では、`flowlet` で公開用のダミーデータ方針を整理したときに考えたことを書きます。

## 先に整理したかったこと

今回まず決めたかったのは、「何を公開してよくて、何を公開してはいけないか」を曖昧にしないことでした。

直感的には分かっていても、運用ルールがないと次のような事故が起こりやすくなります。

- 実在する銀行名や口座名がそのまま seed に残る
- 取引メモに生活情報が混ざる
- README や記事のスクリーンショットに個人情報が映る
- 公開用とローカル用の境界が曖昧になる

個人開発では気を抜きやすい部分ですが、公開前にここを明文化しておく方が安全です。

## flowlet で決めた基本方針

`flowlet` では、公開用データについて次の方針を置きました。

1. 実在の銀行名、口座名、用途名は公開しない
2. 金額、日付、メモ、ID などの具体値も公開用に置き換える
3. seed や fixture は「説明用データ」として割り切る
4. ローカル専用データは公開リポジトリに残さない

大事だったのは、「少しぼかせばよい」ではなく、「公開用として最初から別物にする」と決めたことです。

## 口座 seed をローカル専用にした理由

実際に見直したのは、口座マスタ用の seed データです。

開発初期は、手元で動作確認しやすいように seed を置きたくなります。  
ただ、個人向けアプリでは、そこに実運用に近いデータが入りやすいです。

そのまま運用すると、次の問題があります。

- 開発のしやすさのために置いたデータが、そのまま公開資産になる
- ダミーのつもりでも、実際の運用を推測できる情報が残る
- 今後 CSV や取引データに広がったときに、漏れやすい構造になる

そこで `flowlet` では、実運用寄りの seed をリポジトリから外し、ローカル専用として扱う方針に寄せました。

これは地味ですが、公開前に慌てて探して消すよりずっと安全です。

## ダミーデータを作るときに気をつけたいこと

ダミーデータというと、名前だけ変えればよいように見えます。  
ただ実際には、次の情報もかなり強く個人性を持ちます。

- 金額の傾向
- 支払い日の並び
- メモの書き方
- スクリーンショットに映る一覧構成
- CSV サンプルの列や内容

そのため、公開用データは次のように扱うのがよいと考えました。

- 銀行名は一般化した名称にする
- 用途名は説明しやすい汎用例に置き換える
- 金額は実額から十分に離す
- 日付は現実の生活リズムと結びつきにくい形にずらす
- メモは個人情報を含まない短い説明にする

つまり、匿名化というより、デモ用に再設計する感覚に近いです。

## 公開対象をコードだけにしない

今回整理していて気づいたのは、注意すべきなのはコードや DB seed だけではないという点です。

公開時には、次のものも同じ基準で見直す必要があります。

- 記事に載せる画面キャプチャ
- README に貼る説明画像
- セットアップ手順のサンプル値
- 動作確認用の CSV
- issue やコミットに書いた具体例

アプリ本体はきれいでも、周辺資料に実データが残ることは普通にあります。  
ポートフォリオ用途を考えるなら、ここまで含めて「公開データ方針」として扱った方がよいと感じました。

## ポートフォリオとしての見せ方との両立

実データを消すと、アプリの説得力が弱くなるのではないかという不安もありました。

ただ、むしろ逆で、方針が整理されている方が説明しやすいです。

- なぜこのデータなのか
- なぜ実データを置かないのか
- どこまでを公開用に調整しているのか

このあたりを言語化できると、単にアプリを作っただけでなく、公開と運用まで考えていることが伝わります。

個人開発でも、こうした判断はポートフォリオとして十分アピール材料になると感じました。

## おわりに

`flowlet` では、公開用のダミーデータを「あとで整えるもの」ではなく、早い段階で方針化するものとして扱うことにしました。

実データを避けるのは当然として、そのうえでデモとして分かりやすいデータをどう作るかまで考えると、公開準備はかなり進めやすくなります。

今後は、CSV サンプルやスクリーンショット、README の例示もこの方針にそろえていく予定です。
