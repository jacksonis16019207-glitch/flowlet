# flowlet 要件定義

## 背景

ユーザーは複数の銀行口座を使い分けて生活費、カード引き落とし、投資、貯金を管理している。
以前は銀行側の目的別管理機能を使って用途ごとに残高を分けていたが、口座運用の見直しで資金を集約した結果、目的別残高の見通しが失われた。
また、各種カードや支払いの引き落とし額に対してメイン口座残高が足りているかを都度電卓で確認しており、手間が大きい。

## 何に困っているか

- 1つの実口座にまとまった資金を、目的別残高として把握しづらい
- 現在の残高を把握するために、入出金や振替を頭の中や手計算で追う必要がある
- カード引き落とし予定額に対して、メイン口座残高が足りるかの確認が面倒
- 過去の記録を残したいが、まずは直近の給料日以降を起点にしたい

## 作りたいアプリ概要

`flowlet` は、実在する銀行口座とアプリ内の目的別口座を分けて管理し、資金の配置状況と引き落とし余力を見える化する家計・資産管理アプリである。

実口座の残高と取引履歴を管理しつつ、アプリ上では目的別の仮想口座を持てるようにする。これにより、銀行側で目的別口座がなくても、用途別の残高を維持して把握できる。

## 対象ユーザー

- 主ユーザーはアプリ作成者本人
- 個人利用を前提とし、当面はマルチユーザー対応を行わない

## 現在の口座構成

### メイン銀行

- メイン口座
  - 給与受取
  - 主なカード引き落とし
  - 日常の主な支払い口座
- 使い分け口座
  - 趣味用
  - 健康用
  - 定期的にメイン口座から一定額を振替
  - 利用時はメイン口座へ戻す

### 決済用銀行

- 一部カードの引き落とし口座
- メイン口座から決済用残高へチャージし、送金に使う運用がある

### 貯蓄用銀行

- 普通預金
- 貯蓄用預金
- 投資用、貯金用の資金を保持
- 定期的に普通預金へ入金し、貯蓄用預金へ振替
- 利用時は貯蓄用預金から普通預金へ戻し、メイン口座へ振込

### 以前の管理単位

- 目的別口座
  - 生活イベント
  - 定期支払
  - 特別費
  - 趣味・自己投資
- 投資用預金

## 解決したい課題

### 課題1: 目的別残高の見える化

実口座は1つでも、アプリ上では複数の目的別口座を持ち、用途ごとの残高を常に確認できるようにしたい。

### 課題2: 現在残高の再現

入出金、振替、口座間移動を記録し、現在の実口座残高と目的別残高を再現できるようにしたい。

### 課題3: 引き落とし不足の事前確認

カードや定期支払いの引き落とし予定額を登録し、特定日時点で対象口座の残高が足りるかを事前に確認したい。

## 主要ユースケース

### ユースケース1: 目的別口座で残高を管理する

- ユーザーは実口座を登録する
- ユーザーは目的別口座を作成する
- ユーザーは実口座内の資金を目的別口座へ配分する
- ユーザーは目的別の残高と未配分残高を確認する

### ユースケース2: 取引を記録して残高を更新する

- ユーザーは入金、出金、振替、口座間移動を記録する
- ユーザーは必要に応じて目的別口座とのひも付けを設定する
- アプリは残高を更新する

### ユースケース3: 引き落とし余力を確認する

- ユーザーはカードや固定費の引き落とし予定を手入力で登録する
- アプリは対象口座ごとの引き落とし予定合計を算出する
- アプリは残高不足の有無と不足見込み額を表示する

### ユースケース4: 過去明細を読み込む

- ユーザーは CSV ファイルを取り込む
- アプリは直近の給料日以降の履歴を投入する
- ユーザーは初期残高再現のために補正入力を行える

## 機能要件

### 実口座管理

- 銀行口座だけでなく、資産・負債の管理対象を登録できる
- 管理対象は少なくとも銀行口座、クレジットカード、現金、電子マネーを扱える設計にする
- 提供元名、管理名、管理対象区分、残高区分を登録できる
- 口座をアクティブ、非アクティブで管理できる
- 将来的にどの口座でも引き落とし管理対象にできる設計にする

### クレジットカード管理

- クレジットカードを管理対象の 1 種類として登録できる
- クレジットカードごとに引き落とし元口座を設定できる
- クレジットカードごとに締め日と支払日を設定できる
- 支払日が土日祝に当たる場合の補正ルールを保持できる
- カード利用明細と請求確定額は分けて扱える設計にする

### 目的別口座管理

- 目的別口座を作成、編集、停止できる
- 目的別口座は1つの実口座に属する
- 目的別口座の例
  - 生活イベント
  - 定期支払
  - 特別費
  - 趣味・自己投資
  - 投資用
  - 趣味用
  - 健康用
- 目的別口座ごとの現在残高を計算できる
- 実口座ごとに未配分残高を計算できる
- 未配分残高と目的別口座間で資金を配分できる
- 目的別口座間で資金を再配分できる
- 配分は複数の目的別口座へ一括で行える
- 配分時は金額指定と割合指定の両方に対応できる設計にする

### 取引管理

- 入金、出金、振替、口座間移動を記録できる
- 通常取引と振替取引を分けて扱える設計にする
- 振替は 1 回の操作で出金側と入金側の 2 明細を記録できる
- 取引日、金額、対象口座、説明、メモを保持できる
- 取引に大分類カテゴリを必須で設定できる
- 取引に小分類カテゴリを任意で設定できる
- カテゴリとサブカテゴリをユーザーが登録、編集、削除できる
- 取引で参照中のカテゴリやサブカテゴリを削除した場合は、履歴保護のため停止状態に切り替える
- 必要に応じて目的別口座との関連付けができる
- 残高計算の基礎データとして利用できる
- クレジットカード利用も取引として記録できる
- 振替・振込時に同時に配分を登録できる UI を提供できる設計にする

### 引き落とし予定管理

- 引き落とし予定を手入力で登録できる
- 保持項目
  - 支払い名
  - 金額
  - 引き落とし日
  - 対象口座
  - 支払い種別
  - 繰り返し有無
- 対象口座ごとの引き落とし予定合計を確認できる
- 対象口座の残高と比較して不足の有無を判定できる

### CSV 取込

- 銀行明細 CSV を取り込める
- まずは対象フォーマットを限定してよい
- 直近の給料日以降の明細取り込みを初期目標とする
- 取込後に補正や分類を手動で行える

### ダッシュボード

- 実口座ごとの現在残高を表示する
- 目的別口座ごとの現在残高を表示する
- 未配分残高を表示する
- カテゴリ別の収支を将来的に集計できる設計にする
- 近い将来の引き落とし予定と不足見込みを表示する

## 非機能要件

- 個人利用前提で、まずはローカル環境とローカル本番環境で動作する
- monorepo 構成を採用する
- backend は Java + Spring Boot
- frontend は React + Vite + TypeScript
- DB は PostgreSQL
- Docker でローカル起動できる
- frontend の build 成果物は backend の static 配下に配置して配信する
- PC、スマホ、タブレットで利用しやすい UI にする
- iPhone、iPad から SSH 接続で作業できるよう、ローカル運用を意識する

## MVP 範囲

### Must

- 実口座の登録
- クレジットカード口座の登録
- 目的別口座の登録
- 取引の手入力
- 振替・振込の入力
- 目的別口座への配分
- 目的別残高の表示
- 未配分残高の表示
- 引き落とし予定の手入力
- 対象口座の引き落とし不足判定
- CSV 取込の初版

### Should

- 口座間移動の扱いをわかりやすくする
- 月次の入出金集計
- 目的別口座への配分変更履歴

### Could

- 複数口座を横断した引き落とし余力表示
- CSV フォーマット追加対応
- カテゴリ分類の自動補助
- グラフ表示

### Won't for now

- 銀行 API 連携
- 自動同期
- マルチユーザー対応
- 本格的な家計簿分析機能

## 想定ドメインモデル

- `Account`
  - 資産・負債の管理対象
- `GoalBucket`
  - 目的別口座
- `GoalBucket` MVP
  - 1つの`Account`に属する目的別口座として扱う
  - 項目は`goalBucketId`, `accountId`, `bucketName`, `active`, `createdAt`, `updatedAt`
  - 停止は`active`で管理する
- `Transaction`
  - 入出金、振替、移動、カード利用
- `GoalBucketAllocation`
  - 未配分残高と目的別口座残高の配分変更
- `Category`
  - 取引の大分類
- `Subcategory`
  - 取引の小分類
- `ScheduledPayment`
  - 引き落とし予定
- `BalanceSnapshot`
  - 任意時点の残高確認用

## 将来拡張

- どの実口座でも引き落とし管理ができるようにする
- 取引 CSV から引き落とし候補や定期支払い候補を抽出する
- 複数年の履歴分析を行う
- 投資口座や資産クラス別の見える化を追加する
- 記事化しやすいように、開発ログや判断理由を docs に蓄積する

## 成功条件

- 現在の各実口座残高をアプリで把握できる
- 目的別残高を銀行の機能に依存せず維持できる
- メイン引き落とし口座の残高不足を事前に把握できる
- 直近の給料日以降の記録を継続して残せる
